# Product Brief

## Problem

需要建设一个面向内部使用的大模型 API 分发平台，统一接入多个上游 provider，向下提供兼容 OpenAI 的代理接口和平台自定义增强接口，并配套一个可登录的管理后台以及完整的商业闭环能力。

当前痛点是多家 provider 的接入方式、鉴权方式、模型命名、错误格式、限流规则和计费口径都不一致，导致调用侧、运营侧和配置侧成本高，难以统一管理与扩展。

## User

平台维护者与内部业务接入方。

## Goal

规划并分阶段建设一个内部可用的大模型 API 分发平台，第一轮输出清晰的架构设计、阶段拆分和实现计划，随后逐步完成核心代理层、管理后台和商业系统。

## Non-goals

- 当前任务不要求一次性上线所有 provider 的全部高级特性。
- 当前任务不以多租户 SaaS 为目标。
- 当前任务不优先覆盖每个厂商私有的全部非标准接口。

## Success Criteria

- 能清晰定义平台边界、子系统拆分和推荐的实施顺序。
- 能明确 MVP 与后续阶段的功能归属，避免一次性堆叠全部复杂度。
- 产出后续可执行的设计文档和实现计划。

## Constraints

- 默认使用 AI Dev Team Starter Kit 标准流程推进。
- 面向内部使用，不以第一期多租户隔离为前提。
- 需要覆盖 `OpenAI-compatible API` 与平台增强 API。
- 需要包含可登录的管理后台。
- 需要考虑商业闭环，包括套餐、充值、订单、支付、余额、账单。
- 目标 provider 包括 `OpenAI`、`Anthropic`、`Gemini`、`DeepSeek` 以及小米等大众平台。

## Open Questions

- 第一版建议拆分为几个实施阶段，以及每阶段的交付边界。
- 平台优先支持哪些上游能力：文本、推理、多模态、Embedding、图像生成、语音等。
- 商业系统第一版要接入哪些支付方式。
- 管理后台是否需要区分管理员、运营、财务等角色权限。
