# 燃刻移动端减法重排设计

## Summary

本轮把“燃刻”移动端从信息偏多的记录工作台，收敛成日常执行型 App。核心方案采用用户确认的 A 方案：减法重排版。

首页只回答“今天最该补什么”，其它页面只保留“今天 + 最近 3 条 + 查看全部”。复杂功能不删除，只从主界面下沉到设置的高级功能入口。数据结构、store 持久化 key、备份 JSON、后端接口和 APK 包名保持不变。

## Goals

- 降低移动端第一屏复杂度，让用户打开 App 后立刻知道下一步该做什么。
- 首页只展示一个主任务和一句轻量摘要，不再铺执行清单、趋势、建议和多张状态卡。
- 饮食、训练、身体、趋势页面默认只展示当天结果、最近 3 条记录和查看全部入口。
- 底部 `+ 记录` 改成智能优先，默认打开今天最该补的记录类型。
- 食物库、训练模板、本周安排、完整历史筛选、导入导出等复杂能力统一放到设置的高级功能里。

## Non-Goals

- 不重做桌面端。
- 不修改业务类型，例如 `Food`、`MealEntry`、`WorkoutSession`、`BodyEntry`、`RecoveryEntry`。
- 不修改 localStorage key、备份 JSON 结构、后端接口、Android 包名。
- 不删除食物库、训练模板、周计划、历史筛选等能力。
- 不引入新的视觉主题，只在现有白底和燃橙重点色基础上做减法。

## Confirmed Product Choices

### 首页优先级

首页只问一件事。主任务按以下顺序计算：

1. 饮食没记够三餐。
2. 体重没记。
3. 恢复没记。
4. 训练没记。

饮食优先，因为它是最高频记录；体重和恢复成本低；训练不是每天必须。

### 首页摘要

主任务下方只保留一句话摘要，例如：

```text
已吃 620 kcal，训练未记
```

不展示多列数字、不展示多张状态卡。

### 页面默认复杂度

饮食、训练、身体、趋势四个业务 tab 默认采用：

```text
今天 + 最近 3 条 + 查看全部
```

完整历史、模板、数据库和筛选能力不作为默认第一屏。

### `+ 记录` 行为

底部 `+ 记录` 点击后采用智能优先：

- 默认进入首页当前主任务对应的记录流程。
- 弹层仍提供饮食、训练、体重、恢复其它入口，方便用户切换。
- 不再让四个入口成为第一视觉负担。

## Mobile UX Design

### 首页

首页第一屏结构：

```text
今天
[日期切换]

你现在最该做：
补早餐 / 记体重 / 记恢复 / 记训练

一句摘要：
已吃 620 kcal，训练未记

[去记录]
```

首页不再默认展示：

- 多项执行清单。
- 连续记录卡片。
- 多张恢复、趋势、建议状态卡。
- 复杂周报或图表。

这些信息可以下沉到对应 tab 或趋势页。

### 饮食页

默认回答“今天吃了什么”。

- 显示早餐、午餐、晚餐、加餐四块。
- 每块展示食物名、总热量和必要的营养摘要。
- 无记录时显示轻量空状态和记录入口。
- 下方只露最近 3 条饮食记录。
- `查看全部` 进入完整历史或高级功能。
- 食物库不在主界面出现，只作为搜索录入、常吃带入、拍照估算候选和营养值填充的数据源。

### 训练页

默认回答“今天有没有练”。

- 显示今天训练状态。
- 已训练时展示训练名、时长、消耗和动作摘要。
- 未训练时给一个轻量记录入口。
- 下方只露最近 3 条训练。
- 模板、周计划、模板编辑和完整历史下沉到查看全部或设置高级功能。

### 身体页

默认回答“今天身体状态如何”。

- 显示体重、恢复、睡眠、步数、喝水的今日值或未记录状态。
- 提供记录今天身体状态的入口。
- 下方只露最近 3 条身体相关记录。
- 历史筛选和完整列表下沉到查看全部。

### 趋势页

默认回答“这周整体怎么样”。

- 第一屏展示周报总结。
- 保留最近变化、完成度和一个建议关注点。
- 不默认铺多图表报表。
- 详细图表和更长周期分析放到展开区或查看全部。

### 设置高级功能

设置页保留日常设置，并新增或整理为高级功能入口：

```text
设置
- 个人目标和数据
- 高级功能
  - 食物库和常用食物
  - 训练模板
  - 本周训练安排
  - 完整历史记录
  - 数据导入导出
  - 清空记录 / 恢复示例
```

高级功能只是入口重排，不改变功能读写方式。

## Components And Boundaries

- `FitnessApp` 继续负责 tab、日期、设置、记录弹层和移动端壳层。
- `TodayView` 收敛为主任务、摘要和记录入口。
- `MealsView` 默认渲染今日三餐加加餐、最近 3 条和查看全部入口。
- `WorkoutsView` 默认渲染今日训练状态、最近 3 条和查看全部入口。
- `BodyView` 默认渲染今日身体状态、最近 3 条和查看全部入口。
- `InsightsView` 默认渲染周报总结、变化和关注点。
- 食物库、训练模板、周计划、完整历史相关组件可复用现有代码，但入口从主视觉移动到设置高级功能或查看全部。

## Data Flow

- 今日主任务使用现有 store 和 selector 计算，不新增业务数据结构。
- 首页摘要从现有饮食、训练、身体和恢复数据派生。
- `+ 记录` 使用同一套主任务判断结果决定默认记录类型。
- 其它记录入口继续调用现有保存逻辑。
- 设置高级功能只改变导航和展示，不改变数据读写。

## Error Handling

- 如果某类数据缺失，页面展示轻量未记录状态，而不是空白。
- 如果今日主任务无法从数据中判断，默认优先进入饮食记录。
- 如果高级功能入口对应数据为空，沿用现有空状态。
- 在线食物查询失败时不影响本地食物搜索和手动记录。

## Testing Strategy

自动化测试重点：

- 首页只展示一个主任务和一句摘要。
- 首页主任务优先级符合：饮食、体重、恢复、训练。
- `+ 记录` 默认打开主任务对应记录流程。
- 饮食页默认能看到早餐、午餐、晚餐、加餐，以及最近 3 条或空状态。
- 饮食页主界面不再暴露食物库管理区，但录入时仍可搜索本地食物。
- 训练页主界面不再暴露模板管理和周计划管理，但仍可通过高级入口进入。
- 身体页主界面不再暴露完整历史工作台，但仍可查看最近记录和进入查看全部。
- 趋势页第一屏突出周报总结，而不是多图表报表。
- 五个 tab 正常切换。

验证命令：

```powershell
pnpm lint
pnpm test
pnpm build
```

需要重新产出 APK 时继续运行：

```powershell
pnpm run android:build:debug
```

## Acceptance Criteria

- 移动端首页第一屏看起来像日常执行 App，不像后台工作台。
- 用户打开 App 后，能在 3 秒内知道今天下一步该补什么。
- 每个主业务 tab 的默认第一屏都只聚焦当天结果。
- 复杂能力仍存在，但默认不干扰日常记录。
- 所有现有业务数据、备份和 Android 包信息保持兼容。
